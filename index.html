Cyber Sécurité

Ces derniers temps, les piratages semblent être de plus en plus répandus sur internet et la sécurité est devenu un point essentiel. Peu importe si vous êtes un petit joueur ou une grande entreprise telle que Linkedin ou Facebook ! Si vous développez des applications Web, il est fort probable que celles-ci soit vulnérables aux attaques.

Vous ne savez pas pourquoi quelqu'un pourrait attaquer votre application ?

Les attaques peuvent avoir de nombreuses motivations, de quelque chose d'aussi simple que d'obtenir un produit ou un service gratuitement à de l'espionnage d'entreprise, ou encore du chantage à l'échelle industrielle.

Les pirates peuvent chercher à voler des codes d’accès ou acquérir les données et informations financières de vos utilisateurs. Ils peuvent rechercher des informations compromettantes ou voler des secrets commerciaux.

Mais comment être moins vulnérable aux attaques ?

Une vulnérabilité d'application est une faiblesse qui peut être exploitée pour compromettre une application. Ces attaques ciblent la confidentialité, l'intégrité ou la disponibilité (appelée «CIA Triad») d'une application, de ses développeurs et de ses utilisateurs.

Il existe un grand nombre de faiblesses dans les applications Web. Cependant, la meilleure source vers laquelle se tourner est le Top 10 de l’OWASP 2017.

Ce top 10 est souvent considéré comme un standard (non officiel), en présentant la vulnérabilité, un scénario d’attaque et comment s’en prémunir. On peut donc le considérer comme un guide de bonnes pratiques à respecter pour protéger notre application des vulnérabilités les plus courantes.

Nous allons donc voir dans les chapitres suivants comment ce prémunir de ces attaques, sécuriser au maximum notre application contre les pirates et pouvoir garder nos informations bien au chaud dans notre base de données.

Sécuriser une application PHP (Symfony, Laravel)

Comme beaucoup de frameworks, Symfony et Laravel comprennent des fonctionnalités de sécurité plutôt robuste par défaut, mais ce serait une erreur de s’appuyer entièrement sur cette réputation pour assurer la sécurité d’une application. Parce que chaque application est différente, le contexte de sécurité est différent lui aussi.

Les développeurs ont donc toujours un rôle important à jouer pour s’assurer que tout est bien configuré. Filtrage des inputs, authentification, gestion des sessions… Cette série d’articles aborde les différentes mécaniques que tout développeur PHP peut utiliser, afin d’assurer la sécurité et la fiabilité de l’application qu’il développe.

1. Protéger votre application contre les injections

Comme pour tout framework ou langage de programmation, la règle numéro 1 est la validation des inputs. Vérifiez que les données fournies par l'utilisateur dans vos formulaires soit bien celles attendues.

Symfony propose une façon simple de mettre en place des contraintes, et de créer des contraintes personnalisées.

  • Contraintes de validation (“validation constraints”) : http://symfony.com/doc/current/reference/constraints.html
  • Créer des contraintes personnalisées : http://symfony.com/doc/current/cookbook/validation/custom_constraint.html

Sur Laravel regarder si la validation est bien configurer : https://laravel.com/docs/5.7/validation


Concernant la base de données, utiliser les mécaniques proposées par les frameworks est une bonne pratique. L'utilisation d'un ORM (Doctrine sur Symfony et Eloquent sur Laravel) vous protége pour les requêtes “classiques”. Évitez d’utiliser des requêtes brutes pour empêcher l’injection SQL.

2. Gérer l’authentification de manière sécurisée

Une authentification bien gérée est primordiale pour le bon fonctionnement de votre application et s'assurer que chacun de vos utilisateurs est accés au contenu souhaiter.
Les documentations fournissent déjà beaucoup d'éléments à respecter, vous pouvez donc vous référer directement à la section Authentification :

  • Symfony Security : https://symfony.com/doc/current/security.html
  • Laravel : https://laravel.com/docs/5.7/authentication

Assurez-vous que les mots de passe sont hashés dans la base de données, avec bcrypt sur Symfony par exemple. Laravel gére déjà le hash à la création d'un nouvel utilisateur.

Quelques bonnes pratiques du quotidien

Gestion des mots passe

On le répète souvent mais c'est pour une bonne raison, quelques pratiques sont à observer dans la gestion de vos mots de passe sur internet

  • Ne pas réutiliser le même mot de passe pour plusieurs comptes
  • Ne pas utiliser des mots de passes similaires pour plusieurs comptes :"PassWord43#" n'est pas assez différent de "paSSworD43$")
  • Utiliser un mot de passe fort (nombre de caractères > 8, minuscules/majuscules, nombres, caractères spéciaux)

Pour choisir vos mots de passe, sachez qu'il existe de nombreux moyen d'en générer aléatoirement, tapez "password" dans DuckduckGo pour voir :)

Sachez aussi qu'il existe pléthore de gestionnaires de mots de passe, ces derniers vous permettent de stocker tous vos mots de passe dans un coffre-fort verrouillé. Vous n'aurez plus qu'à retenir un seul password, celui qui gère votre coffre-fort. J'utilise personnellement LastPass (qui me permet avec son extension de pré-remplir les champs des sites qu'il reconnait) mais n'hésitez pas à fouiller pour trouver le gestionnaire qui conviendra le mieux à vos besoins

Du bon usage du .gitignore

Pour toute information sensible (connection à une base de données, clefs api...) GitHub met à disposition de ses utilisateurs un système pour ne pas suivre les fichiers qui contiennent ces informations.

Il vous suffit pour cela de créer un fichier .gitignore et d'y placer le chemin relatif vers fichiers/dossiers que vous ne voulez pas faire apparaître sur GitHub.

Typiquement, on verra aparaître dans le .gitgnore d'un projet Nodejs la ligne : /node_modules

Déployer sur Heroku

Le déploiement sur Heroku, ce moment tant attendu mais souvent source de nombreuses frustrations...

Je ne parlerai pas ici du déploiement en lui-même mais plus particulièrement des variables d'environnement d'heroku.
Les variables d'environnement ce sont toutes les variables que vous utilisez pour définir l'environnement dans lequel l'application va tourner (quel type de serveur, à quelle base de données se connecter...)

Comme pour Github, il est conseillé de ne pas laisser les fichiers contenant ces informations dans le dossier de déploiement. Du coup il va vous falloir indiquer d'une autre manière à heroku quelles sont ces variables ainsi que leurs valeurs. Vous avez pour cela deux possibilités :

  • Aller dans les paramètres de votre projet sur heroku et annoncer vos variables dans ce qu'heroku appelle les "config vars"
  • Utiliser heroku-cli pour définir directement vos variables via le terminal de commande

SafeSearch The Web

Ne prenez rien pour acquis et réfléchissez aux outils que vous utilisez: même quelque chose d'aussi anodin que faire une recherche sur Google est déjà un choix. De nombreux moteurs de recherches proposent aujourd'hui des services performant tout en protégeant vos données.
N'hésitez pas à varier les outils selon le contexte, chacun a ses spécificités. Pour garder le même exemple: j'utilise Duckduckgo pour toute recherche en anglais ou concernant le dev mais j'utiliserais sûrement Qwant ou Lillo pour rechercher une information en France.

Plus généralement

Gardez vos logiciels à jours pour les mise à jours de sécurité (ceci concerne aussi les environnemet Linux, Mac ou Window), ne cliquez pas n'importe où (internet, mails), vérifiez l'origine d'un programme avant de l'installer, ne stockez pas vos mots de passe sur votre machine, ne laissez pas de session ouverte...

Bref, faites preuve de bon sens !!
